I. Introduction

Package testing on a Linux distribution involves a series of cumbersome processes right from start to finish. When the tester/ developer needs some data from the end user's machine, there are a myriad of issues involved:
  • What data is to be requested?
  • How will the user gather that data?
  • Is it technically feasible for the user to get the information the tester needs (different levels of expertise)?
  • How to reduce the effort of gathering data for testing without compromising on the quality of the data to be collected?
  • After the user has gathered all the data, how is the data sent back to the tester?
With the Pardus Package Testing Framework, we intend to counter all the above problems and create a standardized package testing architecture for Pardus Linux.

Here is how it works - the software tester/ package maintainer requires some information from the end user. To do this, the tester creates a test case file which has all the required information on what data is to be gathered and how it is to be gathered. The tester then sends this file to the user, where the file is parsed and the tests are carried out by the framework. The report generated is then sent back to the tester - all this is done with the minimum effort of the tester and the user without compromising on the effectiveness of the testing.

With the implementation of the framework, we intend to resolve all the issues mentioned above:
  • What data is to be requested?
    Specify the package to be tested and other relevant data in the test case file.

  • How will the user gather that data?
    No need to guide the user to get the data; let the framework handle everything for you!

  • Is it technically feasible for the user to get the information the tester needs (different levels of expertise)?
    Again, the framework handles all this effectively. The data it needs is gathered without you the user having to do anything about it.

  • How to reduce the effort of gathering data for testing without compromising on the quality of the data to be collected?
    The user just has to give the command to run the file which was sent by the tester. Rest everything else is handled by the framework thus reducing the effort of both the tester and the user, while ensuring that the testing is carried out as was expected.

  • After the user has gathered all the data, how is the data sent back to the tester?
    The test report is automatically generated and is sent back to the tester or a bug report is filed.

II. Usage

Usage:

python package-testing.py [options] arguments

Options:

-h, --help
 show this help message and exit
-f FILE, --file=FILE
 specify the input XML testcase file for testing (REQUIRED)
-p FILE, --packages=FILE
 specify the input file for custom package processing
-o FILE, --out=FILE
 specify the output file to print the list of packages in the input XML

Options Details:

1.  -f : XML Testcase Input File

The '-f' option is used to specify the input testcase XML file. This option is REQUIRED.

2.  -p : Custom Package Processing

The '-p' option is used to specify an input file which has a list of the specific packages that you want to parse against the testcase input file (specified by the '-f' option).

For example, you want to run the tests only on the packages 'package-1' and 'package-2', the tests for which are present in the input testcase file. To create a custom package input file, create a new file and enter the list of packages, separated by a newline (a whitespace will also do), as shown:
package-1
package-2
Then, pass the '-p' option to the framework:
./package-testing.py -f <testcase-XML-file> -p <package-input-file>
Note that you can only pass a file with the list of packages as an argument to '-p' and not a list of packages; we will fix this in the future (See TODO)

3.  -o : Output Package List

The '-o' option is used to print a list of all the packages present in the testcase input file specified by the '-f' switch.
./package-testing.py -f <testcase-XML-file> -o <output-package-file>
This will output the list of all the packages in the testcase file to the file specified.

*NOTE*: You can either use the '-p' or the '-o' option, but not both. The framework will throw up an error but you have been warned!

III. Creating Test Cases

1. Types of Tests

In the current release of the project (1.0.0), there are four types of tests.

install
 For installing packages on the user's machine
gui
 For testing packages which have a graphical user interface
shell
 For testing packages which are run from the command line
automated
 For automatic testing of packages which run from the command line

Also note that a single test case file can have more than one type of tests. Theoretically, there is no limit on the amount of tests a single file can have.

i. install

The install test is used for installing packages on the user's machine.

Among other uses, this is particularly useful when you want to test a package that is not shipped with Pardus but is available in the repositories.

A test case XML file representation of an install test is:
<testcase test="install" lang="en">
    <package>package-1</package>
    <package>package-2</package>
    <package>package-3</package>
            .
    <package>package-n</package>
</testcase>
This will install package-1, package-2, package-3 ... package-n on the user's machine.

Tag
 Purpose
<package>
 The package to be installed is specified by the text in the package tag
  • Before attempting to install the package, the framework first checks whether the package is already installed. If it is not, then it goes on to check whether the package is available in the repository. If the package is there in the repository, then only does it attempt to install it. So if a package is not available in the repository, the framework will throw up a missing package error.

  • The user should have appropriate permissions for installing the package, otherwise a privilege error is thrown and the installation fails.

  • Also, no actual testing takes place in this test; the purpose of this is to install packages so as to satisfy dependency requirements or that they can be used for testing by another package later.

  • This test runs from the command line.
-

ii. gui

The GUI test is to test packages that have a graphical user interface.

The purpose of the GUI test is to ask the user to perform and observe actions performed on a graphical user interface and then noting the outcome.

A test case XML file representation of a gui test is:
<testcase test="gui" lang="en">
    <package>package-1</package>
    .
    <case>
       <text>Text 1</text>
       <download>Download 1</download>
       <link>Link 1</link>
    </case>
    .
</testcase>
Tag
 Purpose
<package>
 - The package to be tested is specified by the text in the <package> tag
- If the package is not installed, the framework will attempt to install it
  (In case installation is unsuccessful, the tests for this package will be skipped)
<case>
 The <case> text is the root tag for the actual testing to be performed
<text>
 The text of the <text> tag is the text that will be displayed during testing
<download>
 The file specified by the text of the <download> tag will be downloaded to the current working directory of the script
<link>
 The text of the <link> tag will specify the link that is to be opened in a browser.

A GUI testcase can have multiple packages to be tested. For each package, all the case tags will be executed individually.

So if you have two packages, package-1 and package-2 and both are to be tested for the common <case> tags, your test case file will read:
<package>package-1</package>
<package>package-2</package>
<case>
.
</case>
The <download> tag downloads the file specified by the text of tag to the current working get directory of the script.
<download>www.pardus.org.tr/foo.doc</download>
This will download the file foo.doc which can be used for testing.

The text of the <link> tag will be displayed during testing and the user will be asked to open the link.
  • Each GUI test has to have one <case> tag at least.

  • If the package is not installed, the framework will try to install it. If that fails, testing will be skipped and the user will be notified.

  • The GUI test is run from a GUI window.
-

iii. shell

The shell test is to test packages that are run from the command line or that are to be tested through a command line.

The purpose of the shell test is to ask the user to run some commands and note the outcome.

A test case XML file representation of a shell test is:
<testcase test="shell" lang="en">
    <package>package-1</package>  
    <case>
       <text>Text</text>
       <command>Command</command>
    </case>
    .
</testcase>
Tag
 Purpose
<package>
 - The package to be tested is specified by the text in the <package> tag
- If the package is not installed, the framework will attempt to install it
  (In case installation is unsuccessful, the tests for this package will be skipped)
- There is only one package per a shell test
<case>
 The <case> text is the root tag for the actual testing to be performed
<text>
 The text of the <text> tag is the text that will be displayed during testing
<command>
 The text specified by the  <command> tag will be the commands that the user is expected to run on the shell

The GUI and the Shell cases are similar except for:
  • The GUI test is for packages that have a GUI and is used exclusively for testing such packages. The shell tests is for packages that are run through the command line.

  • The GUI test case can test for multiple packages at once, however the shell test can be used for only a single package.
-

iv. automated

The automated test is for used for automated testing, i.e. running a command, getting the output, comparing it with the expected output and noting the result.

The purpose of the automated test is to fully automate packages that run through the command line and produce a fixed and definite output.

A test case XML file representation of a automated test is:
<testcase test="automated" lang="en">
    <package>package-1</package>  
    <case>
       <command>command</command>
       <expected>expected-output</expected>
    </case>
    .
</testcase>
Tag
 Purpose
<package>
 - The package to be tested is specified by the text in the <package> tag
- If the package is not installed, the framework will attempt to install it
  (In case installation is unsuccessful, the tests for this package will be skipped)
- There is only one package per an automated test
<case>
 The <case> text is the root tag for the actual testing to be performed
<command>
 The text specified by the  <command> tag will be the commands that the user is expected to run on the shell
<expected> The expected output of the command that is executed by the <command> tag

The output is compared with the expected output, making this test completely automated. This is the only test that requires no user intervention.

2. The Test Case Writer Script

To enable testers and developers to easily write test case files, the testcasewriter script is used. It generates XML files according to the required schema and makes it easy for testers and developers to generate test case files quickly.

To use the script, just run ./testcasewriter.py from the src/ folder and follow the instructions on the screen.

IV. Technical Information

The Package Testing Framework was programmed in Python 2.6 and PyQt is used for the graphical user interfaces for the gui and the shell tests.

The XML parsing is handled by the lxml library.

Note that the framework runs only on Pardus GNU/ Linux because it makes use of the PiSi (package manager) API.

V. Try it out!

To try out the latest developmental build from the SVN:
$ svn co http://svn.pardus.org.tr/uludag/trunk/playground/gsoc/testing-framework/
To install the framework, issue the following command:
$ sudo python setup.py install
Running install will both build and install the project. In case you only want to build, you can run:
$ python setup.py build
And then run install later.

Creating a source distribution of the project is easy:
$ python setup.py sdist
Then run the framework with the appropriate options.

A sample test case XML file can be found in the doc folder. Try it out!

Requirements:

Python 2.6+
lxml 2.2+ (Available in the Pardus repository)
